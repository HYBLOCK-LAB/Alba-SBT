# Alba-SBT: 알바 경력 증명 시스템 PRD (Product Requirements Document)

**최종 수정**: 2026-04-10  
**버전**: 1.0  
**상태**: 개발 준비 완료

---

## 📋 Executive Summary

### 문제 정의

현재 알바생의 경력 증명 방식은 **파편화, 휘발성, 수동성** 3가지 치명적 문제를 가지고 있습니다.

| 문제점 | 상세 |
|--------|------|
| **파편화** | 고용보험, 통장 입금 내역, 가게 기록이 흩어져 있음 |
| **휘발성** | 4대 보험 미가입 알바는 사장님이 기억하지 않으면 기록이 사라짐 |
| **수동성** | 매번 서류를 직접 요청하고 발급받아야 함 |

또한 플랫폼에서는 **허위 경력 작성, 경력 부풀리기, 경력 세탁** 등으로 인한 신뢰 붕괴 문제가 심각합니다.

### 솔루션

**블록체인 기반 SBT(Soulbound Token) + EAS(Ethereum Attestation Service)를 활용한 자동 경력 증명 시스템**

- ✅ **데이터 기반**: QR, GPS, POS를 통해 객관적 데이터 자동 수집
- ✅ **자동 발행**: 조건 달성 시 EAS가 자동 발행, 사장님 개입 없음
- ✅ **온체인 기록**: 모든 증명이 블록체인에 영구 기록되어 조작 불가
- ✅ **검증 가능**: 누구나 언제든 EAS Explorer로 상세 데이터 추적 가능

## 🎯 목표

1. **알바생**: "성실합니다"라는 말 대신 **"6개월 무지각 인증"**으로 자신의 경력 증명
2. **사장님**: 신입인지 숙련공인지를 신뢰도 높게 판단하고 즉시 채용
3. **플랫폼**: 허위 경력 작성 원천 차단 및 데이터 기반 채용 생태계 구축

## 📱 제품 범위

### 1️⃣ 경력 관리 앱 (React Native)

> **하나의 앱**, 계정 유형에 따라 완전히 다른 UI로 구현

**알바생 계정**
- 🔐 사번 기반 매장 필터링 (특정 매장 출퇴근 QR만 활성화)
- 📍 GPS 검증 출퇴근 (위치 위조 방지)
- 📊 경력 레벨 대시보드 (현재 레벨, 다음 레벨 진행도)
- 🏆 배지 컬렉션 (획득한 SBT 시각화)
- 🔗 포트폴리오 공유 QR (wallet_address 인코딩, 사장님 스캔용)

**사장님 계정**
- 📅 직원 근태 캘린더 (월간 근무 현황, 실시간 출근 여부)
- 🔐 출퇴근 QR 생성 (당일 매장 고유 QR, 유효시간 설정)
- 👥 직원 관리 (등록/삭제/상세 정보 조회)
- ✅ 승급 서명 승인 (Multisig 처리)
- 🔍 지원자 경력 검증 (QR 스캔 → EAS/SBT 신뢰도 리포트 조회)

### 2️⃣ POS 시뮬레이터 (Next.js 웹)

> 핵심 목적: **"누가(staff_id), 어떤 성과(결제/제조)"** 를 정확히 DB에 기록

**주요 기능**
- 📋 메뉴 관리 (메뉴명, 가격, 카테고리 등록)
- 🛒 주문/결제 (메뉴 선택 → 처리자 선택 → 결제 처리)
- 👨‍🍳 제조 리스트 (KDS - Kitchen Display System)
- 📊 데이터 수집 DB (payment_logs, making_logs)

**제외 범위**
- ❌ 실제 금융 거래
- ❌ 실 POS 하드웨어 연동
- ❌ Phase 1에서 블록체인 기능

## 🔗 블록체인 설계 — EAS + SBT 결합 모델

### 핵심 개념

| 레이어 | 기술 | 역할 | 발행 시점 |
|--------|------|------|---------|
| **하부 (팩트)** | EAS | 세부 근태·업무 증명 | 조건 달성 시 자동 |
| **상부 (배지/레벨)** | SBT (ERC-5192) | 최종 커리어 배지 | 레벨 승급 시만 |

### EAS Building Blocks (추가 및 수정 가능)

각 항목이 조건을 충족할 때마다 **독립적인 EAS가 온체인 자동 발행**됩니다.

| EAS 유형 | 이름 | 발급 트리거 | 데이터 필드 | 재발급 |
|----------|------|------------|-----------|--------|
| `EAS_EXP_TIME` | 업종 경력 증명 | 동일 업종 누적 6개월 | 업종, 근무지, 기간 | ✅ 반복 가능 |
| `EAS_SVC_COUNT` | 서비스 숙련도 증명 | 제조/결제 1,000건 | 서비스 종류, 횟수 | ✅ 반복 가능 |
| `EAS_FAITH_ATT` | 성실성 증명 | 3개월 지각·결근 0회 | 출근율, 기간 | ✅ 반복 가능 |
| `EAS_SCHED_RELI` | 신뢰도 증명 | 3개월 스케줄 변경 0회 | 변경 횟수, 점수 | ✅ 반복 가능 |
| `EAS_SUB_SUPPORT` | 협력도 증명 | 대타 근무 지원 5회 | 지원 횟수, 시간 | ✅ 반복 가능 |

### SBT 레벨 승급 요건(초안)

레벨은 **"어떤 EAS를 몇 개 보유했는가"**로 결정됩니다.

| 레벨 | 승급 요건 |
|------|---------|
| 🛡️ **Lv.1 (Beginner)** | 가입 및 첫 출근 시 자동 부여 |
| **Lv.2** | `EAS_EXP_TIME` ≥ 1개 |
| **Lv.3** | `EAS_EXP_TIME` ≥ 1개 + `EAS_SVC_COUNT` ≥ 1개 |
| ⚔️ **Lv.4 (Intermediate)** | `EAS_EXP_TIME` ≥ 3개 (2개 이상 업종) + `EAS_FAITH_ATT` ≥ 3개 + `EAS_SCHED_RELI` ≥ 2개 |
| **Lv.5** | `EAS_EXP_TIME` ≥ 4개 + `EAS_SVC_COUNT` ≥ 2개 + `EAS_FAITH_ATT` ≥ 5개 |
| **Lv.6** | `EAS_EXP_TIME` ≥ 5개 + `EAS_SVC_COUNT` ≥ 3개 + `EAS_FAITH_ATT` ≥ 6개 + `EAS_SCHED_RELI` ≥ 4개 |
| 🧙 **Lv.7 (Expert)** | `EAS_EXP_TIME` ≥ 6개 (3개 이상 업종) +  `EAS_FAITH_ATT` ≥ 8개 `EAS_SCHED_RELI` ≥ 6개|
| **Lv.8** | `EAS_EXP_TIME` ≥ 7개 + `EAS_SVC_COUNT` ≥ 5개 + `EAS_FAITH_ATT` ≥ 9개 + `EAS_SUB_SUPPORT` ≥ 1개|
| **Lv.9** | `EAS_EXP_TIME` ≥ 8개 + `EAS_SVC_COUNT` ≥ 6개 + `EAS_FAITH_ATT` ≥ 10개 + `EAS_SCHED_RELI` ≥ 8개 |
| 👑 **Lv.10 (Master)** | `EAS_EXP_TIME` ≥ 10개 (5년+, 업종 4종 이상) + `EAS_SVC_COUNT` ≥ 10개 + `EAS_FAITH_ATT` ≥ 12개|
|                        + `EAS_SCHED_RELI` ≥ 10개 + `EAS_SUB_SUPPORT` ≥ 2개 |

**업종 카테고리**: F&B / 유통·물류 / 서비스 / 기타 (추가 고려사항)

## 🔄 전체 데이터 파이프라인 (6단계)

### Step 1️⃣ — Raw Data Collection

**근태 데이터**
- 알바생 앱에서 매장 QR 스캔 → GPS 좌표 + 타임스탬프 → Supabase `attendance` 테이블 기록

**성과 데이터**
- POS 시뮬레이터에서 알바생 계정으로 제조/결제 처리 → `payment_logs` / `making_logs` 테이블에 `staff_id` 포함 실시간 적재

---

### Step 2️⃣ — EAS Attestation (팩트 증명 자동 발행)

- Supabase Edge Functions가 **주기적으로** DB 스캔
- EAS 발행 요건 충족 여부 확인 (예: 6개월 경과, 지각 0회 등)
- 조건 충족 시 백엔드가 **플랫폼 서명** → EAS SDK로 Sepolia 네트워크에 증명서 온체인 발행
- **사장님 개입 없음**: 객관적 데이터에 기반한 완전 자동 발행

---

### Step 3️⃣ — Level-up Detection (승급 판독)

- PM이 구현한 **승급 판독 엔진**이 유저 지갑 주소에 연결된 모든 EAS UID 조회
- 복합 조건 비교 (예: "Lv.4 승급 위해 서로 다른 업종 `EAS_EXP_TIME` 3개 보유 여부")
- 조건 충족 시 `Pending Level-up` 상태로 전환
- Multisig 절차 준비

---

### Step 4️⃣ — Multisig 서명 프로세스

- **사장님 서명 (1/2)**: 가스비 없는 EIP-712 오프체인 서명
  - 내용: "이 알바생의 승급 데이터가 사실임을 보증합니다"
  
- **백엔드 서명 (2/2)**: 서버가 사장님 서명값 + 시스템 서명값 묶음

- 완성된 두 서명을 스마트 컨트랙트로 전송

---

### Step 5️⃣ — SBT 민팅

- Solidity 컨트랙트가 `ecrecover`로 두 서명 검증
- 검증 통과 시 ERC-5192 SBT를 유저 지갑으로 발행
- SBT 메타데이터(JSON) → IPFS/Supabase Storage 저장 → `tokenURI` 연결

---

### Step 6️⃣ — Portfolio & Verification

- 알바생 앱의 SBT 지갑에서 획득 배지 시각적 표시
- 다른 사장님이 알바생의 공유 QR 스캔 → EAS Explorer/Etherscan으로 SBT 뿌리 EAS 데이터 즉시 역추적 가능

## 📅 개발 Phases

### Phase 1 — 백엔드 기반 

**범위**: 데이터 수집 + 저장소 구축

- ✅ QR 출퇴근 시스템 + GPS 검증 (매장 GPS ↔ 스캔 GPS 허용 반경 50m 대조, 불일치 시 거부)
- ✅ POS 연동 모듈 (결제/제조 카운팅)
- ✅ 근태 원장 DB 스키마 (결근, 지각, 스케줄 변경)
- ✅ 배지 조건 판정 배치 엔진 (Supabase Edge Functions)

**담당**: 팀A (개발자 A-1, A-2) + 개발자 B

---

### Phase 2 — 블록체인 연동 (약 6주)

**범위**: EAS 자동 발행 + SBT 발급 시스템

- ✅ DID 지갑 연동 (계정 ↔ 지갑 주소 매핑)
- ✅ EAS 온체인 Attestation 발행 (조건 달성 시 자동, PM이 트리거 구현)
- ✅ SBT 컨트랙트 (ERC-5192: mint/burn only, 양도 불가)
- ✅ Multisig 서명 로직 (EIP-712 + `ecrecover` 검증)
- ✅ SBT 메타데이터 JSON → IPFS/Supabase Storage → tokenURI 연결
- ✅ 승급 판독 로직 (PM이 구현)

**담당**: PM + 개발자 C

---

### Phase 3 — 앱 개발 (약 8주)

**범위**: 사용자 인터페이스 구현

- ✅ React Native 경력 관리 앱 (알바생/사장님 화면 분기)
- ✅ POS 시뮬레이터 Next.js 웹 앱

**담당**: 팀A (A-1) + 개발자 B

```
1. 앱 → 지갑 연동 (MetaMask)
2. 매장 사번 입력 → 해당 매장 출퇴근 QR 활성화
3. 출근 시: QR 스캔 + GPS 검증 → 출근 기록
4. 근무 중: POS에서 결제/제조 수행 → 자동 카운팅
5. 퇴근 시: 퇴근 QR 스캔
6. 6개월 경과 후: EAS_EXP_TIME 자동 발행 (온체인)
7. 승급 조건 충족 → 승급 판독 엔진이 감지 → 사장님 승인 요청
8. 사장님 승인 → Multisig 완료 → SBT 발급
9. 앱의 배지 지갑에 SBT 표시
10. 새 일자리 지원 시: 포트폴리오 공유 QR로 경력 증명
```

### 사장님 관점

```
1. 앱  → 지갑 연동 (MetaMask)
2. 직원 등록 (이름, 사번, 입사일, 근무 일정 등)
3. 매일: 당일 매장 고유 QR 생성 → 직원들에게 배포
4. 캘린더에서 월간 근태 현황 확인
5. 직원의 승급 판독 완료 → 앱에서 승인 요청 수신
6. [승인] 버튼 클릭 → EIP-712 오프체인 서명 → SBT 발급 완료
7. 지원자가 포트폴리오 공유 QR을 제시
8. 앱에서 지원자 QR 스캔 → EAS/SBT 신뢰도 리포트 즉시 조회
9. 우수 직원 발견 → 높은 시급 제안
```

## 🔐 주요 기능 상세

### 경력 관리 앱 — 알바생 화면

#### A. 근태 섹션

- **매장 필터링**: 사번 입력 시에만 해당 매장 출퇴근 QR 활성화 (다른 매장 QR 접근 불가)
- **GPS 검증**: QR 스캔 시 기기 GPS 좌표를 서버로 전송
  - 매장 등록 좌표와 허용 반경(예: 50m) 대조
  - 불일치 시 "위치가 일치하지 않습니다" 경고 후 출퇴근 처리 거부
- **출퇴근 기록**: 검증 통과 시 `attendance` 테이블에 저장 (clock_in, clock_out, gps_coords, shop_id)
- **UI**: 월간 캘린더 형태 (지각 날짜는 빨간색, 정상 출근은 초록색)

#### 수집 데이터 명세 — 근태

| 데이터 항목 | 판정 기준 | DB 컬럼 | EAS 연결 |
|-------------|----------|---------|---------|
| **출석** | 예약 출근 시간 ± 허용 범위(예: 10분) 내 clock_in | `attendance.status = 'on_time'` | `EAS_FAITH_ATT` |
| **지각** | 예약 출근 시간 초과 후 clock_in | `attendance.status = 'late'` | `EAS_FAITH_ATT` |
| **결근** | 근무 예정일에 clock_in 없음 (자정 기준 판정) | `attendance.status = 'absent'` | `EAS_FAITH_ATT` |
| **근무 기간** | 특정 매장의 첫 번째 `clock_in` ~ 마지막 `clock_in` 또는 `clock_out` 사이 기간 (월 단위 환산) | `attendance` 테이블에서 집계: `MIN(clock_in)` ~ `MAX(COALESCE(clock_out, clock_in))`, `store_id` 기준 | `EAS_EXP_TIME` |
| **업종** | 매장 등록 시 설정 (F&B / 유통·물류 / 서비스 / 기타) | `stores.category` | `EAS_EXP_TIME` |
| **스케줄 변경 횟수** | 확정된 근무 일정을 근무 당일 N시간 이전에 변경 요청한 횟수 | `schedule_changes` 테이블 COUNT | `EAS_SCHED_RELI` |
| **대타 지원 횟수** | 다른 알바생의 공석 공지에 자발적으로 지원한 횟수 | `substitute_requests` 테이블 COUNT (status='accepted') | `EAS_SUB_SUPPORT` |

#### B. 경력 대시보드

- **상태 조회**: 유저 지갑에 연결된 SBT(최종 등급) + EAS(개별 증명) API 조회 후 렌더링
- **레벨업 프로그레스 바**:
  - 현재 보유 EAS 개수 vs 다음 레벨 필요 요건을 % 진행률로 표시
  - 예: "Lv.4 승급까지 EAS_EXP_TIME 3개 필요 (현재 1개 / 33%)"
- **공유 QR**: wallet_address 인코딩 → QR 코드 상단 배치 (사장님 앱에서 스캔용)

### 경력 관리 앱 — 사장님 화면

#### A. 근태 및 스케줄 관리

- **QR 생성**: 당일 출퇴근용 매장 고유 QR 코드 생성
  - 유효시간 설정 권장 (예: 08:00 ~ 23:00)
  - 유효 시간 외 스캔 시 거부
- **스케줄러**: 캘린더 뷰
  - 전 직원 월간 근무 현황
  - 실시간 출근 여부 (색상: 출근=초록, 미출근=회색)

#### B. 알바 관리 및 승인

- **직원 리스트**:
  - 매장 소속 알바생 등록/삭제
  - 상세 정보 조회 (이름, 사번, 보유 등급 등)
- **승급 승인 대기 목록** (핵심):
  - 백엔드 승급 판독에서 조건 충족한 알바생만 **자동으로** 리스트에 노출
  - 개별 EAS 발행 건은 이 리스트에 뜨지 않음 (사장님이 한번에 처리 가능)
  - [승인] 클릭 → 지갑 연동 → EIP-712 오프체인 서명 수행
- **Multisig 프로세스**:
  - 사장님 서명 (1/2): "이 승급을 인정합니다" 메시지 서명 (가스비 0)
  - 플랫폼이 자동으로 2번째 서명 추가
  - 스마트 컨트랙트로 전송 → SBT 자동 mint

#### C. 채용 검증

- **지원자 QR 스캔**:
  - 지원자의 wallet_address QR을 스캔
  - EAS Explorer를 통해 해당 지갑의 모든 EAS 조회
  - 보유한 SBT 레벨 및 EAS 목록을 "신뢰도 리포트" 형태로 표시
  - 예: "[경험] F&B 3년 | [성실성] 지각 0회 (6개월 × 2회) | [숙련도] 음료 5,000회 | Lv.6 Expert"

### POS 시뮬레이터

#### 주요 UI 섹션

| 섹션 | 역할 | 필수 로직 |
|------|------|---------|
| 메뉴 관리 (Admin) | 메뉴명, 가격, 카테고리 등록 | 신규 메뉴 추가, 기존 메뉴 수정 |
| 주문/결제 (Main) | 메뉴 선택 → 처리자 선택 → 결제 처리 | 처리자 필터링, 결제 금액 계산 |
| 제조 리스트 (KDS) | 주문 내역을 '준비 중' 상태로 표시 | 주문 추가, 완료 마킹 |
| 처리자 선택 모달 | 결제/제조 완료 시점에 현재 근무자 리스트 | Attendance 동기화 필터링 |

#### 핵심 비즈니스 로직

**로직 A — 처리자 필터링 (Attendance Sync)**
```
근무 직원 = attendance 테이블에서 clock_out IS NULL인 레코드만 필터링
→ 처리자 선택 모달에 표시되는 이름 리스트
목적: 매장에 없는 사람이 실적을 올리는 부정행위 방지
```

**로직 B — 결제 프로세스**
```
1. 메뉴 선택 (예: 아메리카노 5000원)
2. [결제] 버튼 클릭
3. 처리자 선택 모달 팝업 → 근무 중인 알바생 선택 (예: "김철수")
4. payment_logs 테이블에 저장:
   - staff_id: "kim-chulsu"
   - amount: 5000
   - product: "아메리카노"
   - type: "PAYMENT"
   - created_at: 현재 시간
5. "김철수"의 결제 횟수 카운트 + 1
```

**로직 C — 제조 프로세스**
```
1. 결제 완료 시 주문이 KDS에 "준비 중" 상태로 추가
2. 알바생이 음료 제조 완료 → [제조 완료] 버튼 클릭
3. 처리자 선택 모달 팝업 → 실제 제조한 알바생 선택 (예: "이영희")
4. making_logs 테이블에 저장:
   - staff_id: "lee-younghee"
   - product_name: "아메리카노"
   - type: "MAKING"
   - created_at: 현재 시간
5. "이영희"의 제조 횟수 카운트 + 1
```

#### DB 테이블 구조(참고용)

개발자 B가 구현할 필수 테이블:

| 테이블명 | 용도 | 필수 컬럼 |
|---------|------|---------|
| `products` | 메뉴 저장소 | `id`, `name`, `price`, `category`, `created_at` |
| `payment_logs` | 결제 실적 수집 | `id`, `staff_id (FK)`, `total_amount`, `product`, `created_at` |
| `making_logs` | 제조 실적 수집 | `id`, `staff_id (FK)`, `product_name`, `created_at` |

## 📊 검증 계획

### Phase 1 검증 (백엔드 완료 후)
- ✅ QR 출퇴근 시나리오 E2E 테스트
  - GPS mocking으로 매장 좌표 시뮬레이션
  - GPS 불일치 시 출퇴근 거부 확인
- ✅ POS 시뮬레이터 결제/제조 흐름 확인
  - 근무 직원만 처리자로 노출되는지 확인
  - payment_logs / making_logs가 정상 저장되는지 확인

### Phase 2 검증 (블록체인 완료 후)
- ✅ Sepolia 배포 후 SBT mint/burn 트랜잭션 검증
- ✅ EAS Attestation 온체인 조회 (EAS Explorer)
- ✅ Multisig 서명 프로세스 E2E (EIP-712 + ecrecover)
- ✅ SBT 메타데이터 JSON 저장 및 tokenURI 연결 확인

### Phase 3 검증 (앱 완료 후)
- ✅ **Full E2E 시나리오 테스트**:
  1. 알바생이 첫 출근 (Lv.1 자동 부여)
  2. 매일 POS에서 결제/제조 수행 (카운팅)
  3. 6개월 경과 후 EAS_EXP_TIME 자동 발행
  4. 승급 조건 충족 → 승급 판독 감지
  5. 사장님이 승인 → Multisig 완료
  6. SBT 발급 → 알바생 앱에 배지 표시
  7. 다른 사장님이 포트폴리오 QR 스캔 → 경력 검증 가능

## 📝 제약사항 & 주의사항

### 기술 제약
- Phase 1에서는 블록체인 기능 미포함 (Phase 2에서 구현)
- POS는 **시뮬레이터**: 실제 금융 거래 불가능
- GPS 검증은 위도/경도 반경 기반 (더 정교한 지오펜싱은 추후)

### 데이터 제약
- EAS 자동 발행은 조건(6개월, 횟수 등)에 기반
- 조건 미달성 시 EAS 발행 불가 (소급 발행 불가)
- SBT는 일반적 NFT처럼 양도 불가 (ERC-5192 특성)

### 운영 제약
- Sepolia는 테스트넷 (향후 메인넷 마이그레이션 필요)
- MetaMask 기반 운영 (다른 지갑은 향후 추가)
