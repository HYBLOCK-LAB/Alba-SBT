# Alba-SBT: 알바 경력 증명 시스템 PRD (Product Requirements Document)

**최종 수정**: 2026-04-29  
**버전**: 1.0  
**상태**: 개발 준비 완료

---

## 📋 Executive Summary

### 문제 정의

현재 알바생의 경력 증명 방식은 **파편화, 휘발성, 수동성** 3가지 치명적 문제를 가지고 있습니다.

| 문제점 | 상세 |
|--------|------|
| **파편화** | 고용보험, 통장 입금 내역, 가게 기록이 흩어져 있음 |
| **휘발성** | 4대 보험 미가입 알바는 사장님이 기억하지 않으면 기록이 사라짐 |
| **수동성** | 매번 서류를 직접 요청하고 발급받아야 함 |

또한 플랫폼에서는 **허위 경력 작성, 경력 부풀리기, 경력 세탁** 등으로 인한 신뢰 붕괴 문제가 심각합니다.

### 솔루션

**블록체인 기반 SBT(Soulbound Token) + EAS(Ethereum Attestation Service)를 활용한 자동 경력 증명 시스템**

- ✅ **데이터 기반**: QR, GPS를 통해 객관적 데이터 자동 수집
- ✅ **자동 발행**: 조건 달성 시 EAS가 자동 발행, 사장님 개입 없음
- ✅ **온체인 기록**: 모든 증명이 블록체인에 영구 기록되어 조작 불가
- ✅ **검증 가능**: 누구나 언제든 EAS Explorer로 상세 데이터 추적 가능

## 🎯 목표

1. **알바생**: "성실합니다"라는 말 대신 **성실성 인증(결근 없음·지각 관리 증명)**으로 자신의 경력 증명

## 📱 제품 범위

### 통합 앱 (React Native)

하나의 앱에서 계정 유형(알바생/사장님)에 따라 서로 다른 홈 화면을 제공합니다.

**[공통 화면]**
- 지갑 연동: MetaMask/WalletConnect 연동
- 계정 유형 선택 (신규 가입 시만): 알바생 / 사장님
- 로그인 후 `account_type` 기반 자동 라우팅

**[알바생 홈]** (account_type = 'worker') — 섹션 2개
- 매장 연결: 여러 매장 코드 등록 → 사장님 승인 완료 매장만 목록 활성화
- 경력 프로필 대시보드: 현재 레벨(SBT 배지 시각화), 레벨업 프로그레스 바, 포트폴리오 공유 QR 버튼

**알바생 — 매장별 랜딩 — 탭 3개**
- QR 스캔: 출/퇴근 등록 (GPS 또는 매장 QR 검증, 불일치 시 거부)
- 근태 관리: 월간 캘린더, 실시간 근무 현황(예약 퇴근 기준 남은 시간)
- 추가 근무: 사장님이 등록한 추가 근무 요청 목록 조회 및 신청

**[사장님 홈]** (account_type = 'manager') — 탭 2개
- 탭 1️⃣ 매장 관리 (등록된 매장 목록 + 매장 등록하기 버튼)
- 탭 2️⃣ 지원자 경력 검증 (포트폴리오 QR 스캔 → EAS/SBT 신뢰도 리포트 즉시 조회)

**사장님 — 매장별 랜딩 — 탭 5개**
- 인사 관리: 알바 등록(매장코드 발급→알바생 입력→사장 최종 승인) / 삭제 / 상세 조회
- 근태 관리: 캘린더 일정, 금일 출근 현황표
- 추가 근무: 추가 근무 요청 생성 (익월까지. 날짜·시간 입력) 및 알바생 신청 건 수락 처리
- 출퇴근 QR 생성: [QR 생성] 버튼 누르면 10초간 QR 표시 → 10초 후 자동 소멸 / 재생성하려면 버튼 재클릭 / 태블릿 앱고정 운영
- 승급 승인: 조건 충족 시 자동 리스트 등록 → Multi-Signature 서명 승인

## 🎬 사용자 흐름 예시

### 공통 초기 흐름

```
1. 앱 실행 → 지갑 연동 (MetaMask)
2. 신규 사용자: 추가 정보 입력 화면
   - 계정 유형 선택 (알바생 / 사장님)
   - 이름 (필수)
   - 이메일 (선택)
   - 전화번호 (선택)
   → 입력 완료 시 users 테이블에 저장
3. 기존 사용자: wallet_address로 account_type 조회 → 자동 라우팅
   - account_type = 'worker'  → 알바생 홈
   - account_type = 'manager' → 사장님 홈
```

### 알바생 관점

```
1. 앱 → 지갑 연동 (MetaMask)
2. 홈 → [매장 연결하기] → 매장코드 입력 → 사장님 승인 대기
3. 승인 완료 → 매장 카드 활성화, 매장별 랜딩 접근 가능
4. 출근: 매장 랜딩 → [QR 스캔 탭] → [출근 QR 스캔] → GPS/QR 검증 → 출근 기록
5. 실시간: 근태 관리 탭에서 예약 퇴근까지 남은 시간 확인
6. 퇴근: QR 스캔 탭 → [퇴근 QR 스캔]
7. 필요 시: 추가 근무 탭에서 사장님이 등록한 추가 근무 요청 확인 → 기존 스케줄 미충돌 시 [신청하기] → 사장님 수락 대기 → 수락 완료 시 근태 관리 탭 캘린더에 즉시 반영
8. 6개월 경과: EAS_EXP_TIME 자동 발행 → 대시보드 업데이트
9. 승급 조건 충족 → 사장님 승인 → Multisig → SBT 발급 → 배지 추가
10. 새 일자리 지원: 홈 대시보드 → [포트폴리오 공유 QR] 제시
```

### 사장님 관점

```
1. 앱 → 지갑 연동 (MetaMask)
2. 매장 등록:
   - 매장명, 카테고리(대분류 → 하위 업종), 위치(GPS), 사업자번호(선택), 연락처 입력
   - 매장코드 6자리 자동 생성
3. 알바생 관리:
   - 생성된 매장코드를 알바생에게 공유
   - 알바생이 앱에 코드 입력 → 사장 앱에서 최종 승인 (오프체인)
4. 일상 운영:
   - 매일: [QR 생성] 버튼으로 당일 매장 고유 QR 생성 → 10초간 유효 후 자동 소멸 → 필요 시 버튼 재클릭으로 재생성
   - 캘린더에서 월간 근태 현황 + 실시간 출근 현황 확인
   - 추가 근무 탭에서 추가 근무 요청 생성 (익월까지. 날짜·시간 입력) → 알바생 신청 건 수락 (선착순)
5. 승급 승인:
   - 직원의 승급 판독 완료 → 승인 리스트에 자동 등록
   - 각 알바생 선택 후 [승인] → EIP-712 오프체인 서명 → SBT 발급
6. 지원자 검증:
   - 지원자의 포트폴리오 공유 QR 스캔
   - EAS/SBT 신뢰도 리포트 즉시 조회 → 채용 판단
```

## 🔐 주요 기능 상세

### 공통 화면

#### 지갑 연동
- MetaMask 또는 WalletConnect로 지갑 연결
- wallet_address 기준 SIWE 인증

#### 추가 정보 입력 (신규 가입 시)
- 계정 유형 선택: [알바생으로 시작] / [사장님으로 시작] → `users.account_type` 저장
- 이름 (필수) → `users.name`
- 이메일 (선택) → `users.email`
- 전화번호 (선택) → `users.phone`
- 모든 항목 입력 완료 시 `users` 테이블에 저장
- 계정 유형은 한 번 설정 후 변경 불가

#### 로그인 후 라우팅
- `account_type = 'worker'`  → 알바생 홈으로 이동
- `account_type = 'manager'` → 사장님 홈으로 이동

---

### 알바생 화면 (account_type = 'worker')

#### 홈

**섹션 1: 매장 연결**
- **[매장 연결하기] 버튼**: 매장코드(6자리) 입력 → 사장님 승인 요청 전송
- **연결된 매장 카드 목록**: 승인 완료 매장만 활성화 / 미승인은 "승인 대기" 표시
- **카드 탭**: 해당 매장의 랜딩 페이지로 이동

**섹션 2: 경력 프로필 대시보드**
- **현재 레벨 + SBT 배지 시각화**: 획득한 배지 이미지를 그리드 형태로 표시
- **레벨업 프로그레스 바**: 현재 보유 EAS 개수 vs 다음 레벨 필요 요건 비율을 % 진행률로 표시
- **[포트폴리오 공유 QR] 버튼**: wallet_address 인코딩 QR 생성 (채용처 검증용)

#### 매장별 랜딩 — 탭 3개

**탭 1: QR 스캔**
- **[출근 QR 스캔] / [퇴근 QR 스캔] 버튼**
- **검증 로직 (필수 구현)**:
  - GPS 좌표 ↔ 매장 등록 좌표 반경 50m 이내 대조
  - OR 사장님 화면에서 생성한 당일 매장 고유 QR 일치 확인
  - 두 조건 모두 불일치 시: "위치/QR이 일치하지 않습니다" 경고 후 출퇴근 처리 거부
- **출퇴근 기록**: 검증 통과 시 `attendance` 테이블에 저장 (clock_in, clock_out, gps_coords, shop_id)

**탭 2: 근태 관리**
- **월간 근태 현황 캘린더**: 정상 출근(초록), 지각(노랑), 결근(빨강) 색상 구분. 추가 근무 수락 건은 해당 날짜에 별도 표시
- **실시간 근무 현황**: clock_in 완료 후 예약 퇴근 시간까지 남은 시간 카운트다운 표시

**탭 3: 추가 근무**
- **추가 근무 요청 목록**: 사장님이 등록한 추가 근무 요청 목록 표시 (날짜·시간·매장 정보)
- **[신청하기] 버튼**: 기존 스케줄과 시간 충돌 여부 자동 판단 → 충돌 시 신청 불가 처리 (필수 구현) → 충돌 없으면 신청 완료 후 사장님에게 전달
- **신청 현황**: 내가 신청한 건의 상태 표시 (대기 중 / 수락됨 / 미선택). 수락됨 상태로 변경 시 근태 관리 탭 캘린더에 즉시 반영


### 사장님 화면 (account_type = 'manager')

#### 홈 — 탭 2개

**탭 1: 매장 관리**
- **매장 목록**: 등록된 매장 표시 (매장명, 카테고리, 최근 출근 현황)
- **매장 등록하기 버튼**: 새 매장 추가 시 클릭
  - 입력 필드: 매장명, 카테고리(대분류 → 하위 업종), 위치(GPS), 사업자번호(선택), 연락처
  - 매장코드 6자리 자동 생성 (예: 103078)
  - 매장코드 복사 버튼으로 알바생 공유 용이

**탭 2: 지원자 경력 검증**
- **포트폴리오 QR 스캔**: 지원자의 wallet_address QR 스캔
- **신뢰도 리포트 즉시 조회**:
  - 보유한 SBT 레벨 및 모든 EAS 목록 표시
  - 예: "[경험] F&B 3년 | [성실성] 지각 2회 이하 (3개월 구간 × 4회 통과) | [추가 근무] 수락 12회 (1개 보유) | Lv.8 Expert"
  - 블록체인에서 직접 조회 (실시간, 조작 불가)

#### 매장별 랜딩 — 탭 5개

**탭 1: 인사 관리**
- **알바 등록**: 생성된 매장코드를 알바생에게 공유 → 알바생이 앱에 입력 → 사장 승인 (오프체인)
- **반복 근무 요일·시간 설정**: 승인 완료 직후 사장님이 해당 알바생의 반복 근무 패턴 입력 (근무 요일 복수 선택, 출퇴근 시각) → 저장 시 당월 잔여일 + 익월 `schedules` 레코드 즉시 자동 생성. 이후 매월 25일 B-1 스케줄러가 미생성 건에 대해 익월 레코드 일괄 생성
- **직원 리스트**: 등록된 알바생 표시 (이름, 현재 등급, 상세 조회 버튼). 상세 조회에서 반복 근무 패턴 수정 가능 (수정 시 미래 날짜의 schedules에만 반영)
- **직원 삭제**: 선택 후 삭제 처리

**탭 2: 근태 관리**
- **일정 캘린더**: 전체 알바생의 월간 근무 현황 (색상 구분). 추가 근무 수락 건은 해당 날짜에 별도 표시
- **금일 출근 현황표**: 실시간 출근/미출근 현황 (색상: 출근=초록, 미출근=회색, 결근=빨강, 지각=노랑)

**탭 3: 추가 근무**
- **[추가 근무 요청 생성] 버튼**: 날짜·시간 입력 후 등록 → 현재 월 + 익월까지만 요청 가능
- **등록된 요청 목록**: 요청 건별 신청 현황 표시 (신청 인원 수, 각 알바생 이름)
- **[수락] 버튼**: 신청 목록에서 수락할 알바생 1명 선택 → 수락 완료 시 사장님·알바생 근태 관리 탭 캘린더에 즉시 반영 + 해당 알바생의 EAS_EXTRA_ACC 집계에 반영 (extra_work_requests 테이블 — 별도 DB 설계 예정)

**탭 4: 출퇴근 QR 생성**
- **[QR 생성] 버튼**: 누르면 당일 매장 고유 QR 표시 → 10초 후 자동 소멸
- **10초 카운트다운**: QR 표시 중 남은 유효 시간 표시
- **재생성**: QR 소멸 후 버튼을 다시 눌러야 새 QR 생성 (자동 갱신 없음)
- **유효 시간 설정**: 예: 08:00 ~ 23:00 (설정 범위 외 스캔 시 거부)
- **태블릿 앱고정**: 매장의 태블릿에 QR을 계속 노출 (가능하도록 UI 최적화)

**탭 5: 승급 승인**
- **승급 조건 충족 리스트**: 백엔드 판독에서 조건 달성한 알바생만 자동 노출
- **각 알바생 선택 후 [승인] 버튼 클릭**:
  - 지갑 연동 → EIP-712 오프체인 서명 (가스비 0)
  - 플랫폼이 자동으로 2번째 서명 추가
  - 스마트 컨트랙트 전송 → SBT 자동 발급

## 🔗 블록체인 설계 — EAS + SBT 결합 모델

### 핵심 개념

| 레이어 | 기술 | 역할 | 발행 시점 |
|--------|------|------|---------|
| **하부 (팩트)** | EAS | 세부 근태·업무 증명 | 조건 달성 시 자동 |
| **상부 (배지/레벨)** | SBT (ERC-5192) | 최종 커리어 배지 | 레벨 승급 시만 |

### EAS Building Blocks 

각 항목이 조건을 충족할 때마다 **독립적인 EAS가 온체인 자동 발행**됩니다.

| EAS 유형 | 이름 | 발급 트리거 | 데이터 필드 | 재발급 |
|----------|------|------------|-----------|--------|
| `EAS_EXP_TIME` | 업종 경력 증명 | 동일 매장에서 6개월 달성 (매장별 독립 판정. 같은 업종이라도 다중 매장 기간 합산 없음. 매장당 EAS 1회 발급) | 업종(대분류), 하위카테고리, 매장명, 근무 기간 | ✅ 반복 가능 |
| `EAS_FAITH_ATT` | 성실성 증명 | 매장별 첫 출근일 기준 90일 고정 구간 내 결근 0회 + 지각 2회 이하 (각 매장 독립 판정. 구간 단위 평가, 90일마다 새 구간 시작. 다중 매장 근무 시 매장마다 별도 EAS 발급 가능) | 결근 횟수, 지각 횟수, 구간 기간 | ✅ 반복 가능 |
| `EAS_WORK_COMP` | 근무 완수 증명 | EAS_FAITH_ATT 조건 충족 전제 (결근 0회 + 지각 2회 이하) + 구간 내 on_time 출근한 모든 근무에서 조기 퇴근 0회 (clock_out ≥ 예정 퇴근 시각. late 출근 건은 퇴근 시각 무관 제외. clock_out 없으면 미완수. 각 매장 독립 판정. 90일마다 새 구간 시작. 다중 매장 근무 시 매장마다 별도 EAS 발급 가능) | 근무 완수 횟수, on_time 출근 횟수, 구간 기간 | ✅ 반복 가능 |
| `EAS_EXTRA_ACC` | 추가 근무 수락 증명 | 사장님이 요청하고 알바생이 수락한 추가 근무 누적 10회마다 1개 발급 (전 매장 합산. 기한 없음. 10회마다 반복 발급. 수락 건만 집계, 거절 건 미포함) | 누적 수락 횟수 | ✅ 반복 가능 |

### SBT 레벨 승급 요건

레벨은 **"어떤 EAS를 몇 개 보유했는가"**로 결정됩니다.

#### 승급 설계 기준

**시간 기반 진입 기준 (순차 고용 기준)**

| 티어 | 레벨 | 최소 달성 시기 |
|------|------|-------------|
| 🛡️ Beginner | Lv.1 ~ Lv.4 | Lv.4: ~18개월 |
| ⚔️ Intermediate | Lv.5 ~ Lv.7 | Lv.5 진입: ~2년 |
| 🧙 Expert | Lv.8 ~ Lv.9 | Lv.8 진입: ~3년 |
| 👑 Master | Lv.10 | ~4년 |

**EAS_FAITH_ATT 효율 설계 (최소 경로 대비 요구 비율)**

`EAS_EXP_TIME` N개 요건 달성을 위한 최소 경로(N매장 × 6개월)에서 획득 가능한 최대 `EAS_FAITH_ATT` 대비 요구 수의 비율.

| 티어 | 목표 효율 | 의미 |
|------|---------|------|
| Beginner | ~50% | 성실성 부담 없이 경험 쌓기 단계 |
| Intermediate | ~70%대 평균 | 꾸준한 출석 필요, 과도한 장벽 없음 |
| Expert | ~80%대 | 열심히 꾸준하게 노력한 자가 달성 가능 |
| Master | ~90%대 | 전체 알바생 상위 10% 수준 |

**EAS_WORK_COMP 설계 원칙**: FAITH_ATT 기간 중 1회의 조기 퇴근 허용 (= FAITH_ATT 요구 수 − 1).  
레벨이 높아질수록 WORK_COMP/FAITH 비율이 50% → 93%로 상승해 Master는 사실상 완벽한 근무 완수를 요구.

|------|-------------|------------|--------------|---------|----------|--------------|-----------|

| Lv.6 | 4 | 24개월 | 8 | 6 | 75% | 5 | 83% |
| Lv.7 | 5 | 30개월 | 10 | 8 | 80% | 7 | 87.5% |
| Lv.8 | 6 (3종) | 36개월 | 12 | 10 | 83.3% | 9 | 90% |
| Lv.9 | 7 | 42개월 | 14 | 12 | 85.7% | 11 | 91.7% |
| Lv.10 | 8 (4종) | 48개월 | 16 | 15 | 93.75% | 14 | 93.3% |

> 티어별 평균 FAITH 효율: Beginner 50% / Intermediate 72.5% / Expert 84.5% / Master 93.75%

## 🔄 전체 데이터 파이프라인 (5단계)

### Step 1️⃣ — Raw Data Collection

| 데이터 항목 | 판정 기준 | DB 컬럼 | EAS 연결 |
|-------------|----------|---------|---------|
| **출석** | clock_in 시각 ≤ 예약 출근 시간 (grace period 없음. 정각 포함) | `attendance.status = 'on_time'` | `EAS_FAITH_ATT` |
| **지각** | 예약 출근 시간 초과 후 ~ +30분 이내 clock_in | `attendance.status = 'late'` | `EAS_FAITH_ATT` |
| **결근** | 예약 출근 시간 +30분 초과 후에도 clock_in 없음, 또는 당일 clock_in 자체 없음 | `attendance.status = 'absent'` | `EAS_FAITH_ATT` |
| **근무 기간** | 특정 매장의 첫 번째 `clock_in` ~ 마지막 `clock_in` 또는 `clock_out` 사이 기간 (월 단위 환산) | `attendance` 테이블에서 집계: `MIN(clock_in)` ~ `MAX(COALESCE(clock_out, clock_in))`, `store_id` 기준, `type='regular'` 및 `type='extension'` 모두 포함. 매장별 독립 판정 (다중 매장 합산 없음) | `EAS_EXP_TIME` |
| **업종** | 매장 등록 시 설정 (F&B / 유통·물류 / 서비스 / 기타) | `stores.category` | `EAS_EXP_TIME` |
| **추가 근무 수락 횟수** | 사장님이 요청하고 알바생이 수락한 추가 근무 건 누적 집계. 거절 건 미포함. clock_out 완료 여부 무관 (수락 의사 기준) | `extra_work_requests` 테이블에서 집계: `COUNT(status = 'accepted')`, `user_id` 기준 (전 매장 합산. store_id 구분 없음. extra_work_requests 테이블은 현재 schema.sql 미포함 — 별도 DB 설계 예정) | `EAS_EXTRA_ACC` |
| **근무 완수** | on_time 출근 건에 한해 실제 퇴근 시각이 예정 퇴근 시각 이상인지 판정. late 출근 건은 평가 제외. clock_out_time이 NULL이면 미완수 처리 | `attendance` JOIN `schedules` on `attendance.schedule_id = schedules.id`: `attendance.status = 'on_time'` AND `attendance.clock_out_time::TIME >= schedules.scheduled_end_time`. 매장별(`store_id`) 독립 집계 | `EAS_WORK_COMP` |

### Step 2️⃣ — EAS Attestation (팩트 증명 자동 발행)

- Supabase Edge Functions가 **주기적으로** DB 스캔
- EAS 발행 요건 충족 여부 확인 (예: 6개월 경과, 지각 0회 등)
- 조건 충족 시 백엔드가 **플랫폼 서명** → EAS SDK로 Sepolia 네트워크에 증명서 온체인 발행
- **사장님 개입 없음**: 객관적 데이터에 기반한 완전 자동 발행

---

### Step 3️⃣ — Level-up Detection (승급 판독)

- PM이 구현한 **승급 판독 엔진**이 유저 지갑 주소에 연결된 모든 EAS UID 조회
- 복합 조건 비교 (예: "Lv.4 승급 위해 서로 다른 업종 `EAS_EXP_TIME` 3개 보유 여부")
- 조건 충족 시 `Pending Level-up` 상태로 전환
- Multisig 절차 준비

---

### Step 4️⃣ — Multisig 서명 프로세스

#### 1. 승급 조건 충족 감지 — B-2 담당
- 승급 판독 엔진이 알바생의 EAS 보유 현황을 주기적으로 조회
- 조건 충족 시 해당 알바생의 `pending_level_up` 상태를 DB에 저장
- 사장님이 앱의 '승급 승인 탭'을 열 때마다 해당 매장의 pending 리스트를 자동 표시 (푸시 알림 없음)

#### 2. 사장님 서명 (sig1) — C 담당
- C가 구현한 '승급 승인 탭'에서 사장님이 **[승인]** 버튼 클릭
- *다중 매장 알바생의 경우: `staff_assignments.approved_at`이 가장 이른 날짜의 매장(주 매장) 사장님이 sig1 서명. 승급 승인 탭은 해당 주 매장 사장님 앱에만 표시.*
- B-2로부터 nonce 발급 받은 후, EIP-712 양식으로 MetaMask에 서명 요청
  - 서명 대상: `{ worker: address, level: uint8, nonce: uint256 }`
  - nonce는 B-2가 발급, 재사용 방지
- 사장님이 MetaMask에서 서명 확인 → sig1 (bytes) 생성
- C가 sig1을 B-2 API로 전송
  - `POST /approvals/sign` → `{ workerAddress, level, nonce, sig1 }`

#### 3. 플랫폼 서명 (sig2) — B-2 담당
- B-2 NestJS 서버가 sig1 수신 후 즉시 플랫폼 서명 생성
- 플랫폼 private key: 환경변수 `PLATFORM_SIGNER_PRIVATE_KEY`에서 로드
- 동일한 EIP-712 양식(`{ worker, level, nonce }`)으로 플랫폼 서명 생성 (ethers.js)
- 유효기간 없음 — sig1 수신 즉시 sig2 생성 후 컨트랙트 호출

#### 4. 컨트랙트 호출 — B-2 담당
- 함수 시그니처 (예시): `approveLevelUp(address worker, uint8 level, uint256 nonce, bytes sig1, bytes sig2, string tokenURI)`
- `tokenURI`: Supabase Storage에 업로드한 메타데이터 JSON의 공개 URL
  - 예: `https://<project>.supabase.co/storage/v1/object/public/sbt-metadata/{workerAddress}/{level}.json`
  - C가 배지 화면에서 SBT를 조회할 때 이 tokenURI를 열어 배지 정보를 표시함
- 컨트랙트가 `ecrecover`로 sig1 = 사장님 주소, sig2 = 플랫폼 주소 검증
- 검증 통과 → ERC-5192 SBT 발행 + tokenURI 온체인 저장

#### 5. 실패 처리
- 서명 검증 실패 또는 컨트랙트 호출 실패 시:
  - 사장님 화면에 에러 메시지 표시
  - **[재시도]** 버튼 노출 → 클릭 시 B-2가 새 nonce 발급 → 처음부터 재서명
  - 기존 sig1은 폐기 (nonce 재발급으로 무효화)

---

### Step 5️⃣ — SBT 민팅

- Solidity 컨트랙트가 `ecrecover`로 두 서명 검증
- 검증 통과 시 ERC-5192 SBT를 유저 지갑으로 발행
- SBT 메타데이터(JSON) → Supabase Storage 저장 → `tokenURI` 연결

---

### Step 6️⃣ — Portfolio & Verification

- 알바생 화면의 SBT 지갑에서 획득 배지 시각적 표시
- 다른 사장님이 알바생의 공유 QR 스캔 → EAS Explorer/Etherscan으로 SBT 뿌리 EAS 데이터 즉시 역추적 가능

#### C ↔ A (지갑 연동 및 web3 화면)

개발자 C는 스마트 컨트랙트와 web3 로직 레이어(훅·서비스)를 담당하고, A는 UI 컴포넌트를 구현합니다.  
C가 인터페이스 명세를 먼저 확정(Phase 3 시작 전)하면, A가 해당 훅·서비스를 UI에 연동하는 방식으로 협업합니다.

| 전달물 | 설명 | 연결 대상 | 시점 |
|--------|------|----------|------|
| **MetaMask/WalletConnect SDK 모듈** | C가 캡슐화한 지갑 연동 공통 모듈 | A-1이 일반 UI 화면에서 지갑 주소 참조 시 사용 | Phase 2 시작 전 |
| **WalletConnect Project ID** | 지갑 연동 SDK 설정값 | 앱 전체 환경 설정 | Phase 2 시작 전 |
| **ABI.json + Contract Address** | 컨트랙트 함수 목록 및 Sepolia 배포 주소 | A가 구현하는 web3 화면 UI에서 C의 훅·서비스를 통해 간접 사용 | Sepolia 배포 직후 |
| **web3 훅·서비스 인터페이스 명세** | C가 제공하는 훅(`useSBTData` 등)·서비스의 입출력 타입 정의 문서. A가 UI 구현 시 API 계약으로 활용 | A가 web3 화면 UI 컴포넌트 구현 시 | **Phase 3 시작 전 (선행 조건)** |

*`useSBTData()` 훅의 데이터 출처: 온체인 직접 조회. SBT 목록은 ERC-5192 컨트랙트 RPC 호출(`balanceOf`, `tokenURI`), EAS 데이터는 EAS SDK로 온체인 조회(UID 기반). 레벨 진행률은 보유 EAS 수 기반 로컬 계산. **B-2는 SBT/EAS 목록 조회 API를 별도 제공하지 않음.***

**EIP-712 서명 양식 예시**

```json
{
  "domain": {
    "name": "AlbaSBT",
    "version": "1",
    "chainId": 11155111,
    "verifyingContract": "0x배포주소"
  },
  "types": {
    "Approval": [
      { "name": "worker", "type": "address" },
      { "name": "level",  "type": "uint8"   },
      { "name": "nonce",  "type": "uint256"  }
    ]
  }
}
```

**승인 흐름 (C + B-2 연계)**

```
사장님이 [승인] 버튼 클릭 (C가 구현한 화면)
        ↓
C: EIP-712 양식으로 MetaMask에 서명 요청
        ↓
사장님 MetaMask에서 "확인"
        ↓
C: 서명값을 백엔드(B-2)로 전송
        ↓
B-2: 플랫폼 서명 추가 후 C가 만든 컨트랙트 호출
        ↓
컨트랙트: 두 서명 검증 통과 → SBT 발행
        ↓
C: 알바생 화면 배지 대시보드에서 ABI로 SBT 조회 → 배지 표시
```

#### C → B-2 (Multisig 서명 인터페이스)

## 📅 개발 Phases

> **Phase 구성 원칙**
> - Phase 0에서 모든 팀원의 선행 조건(블로커)을 사전 해소
> - Phase 1부터 백엔드·블록체인·UI를 **병렬 진행**하여 A의 유휴 시간 제거
> - web3 UI는 C의 훅·서비스 인터페이스 확정 이후 Phase 3에서 통합
>
> | Phase | 핵심 목표 |
> |-------|-----------|
> | Phase 0 | 선행 조건 해소 (JWT 명세, EAS 스키마, 와이어프레임) |
> | Phase 1 | 백엔드 API + 인증 + 일반 UI 병렬 개발 |
> | Phase 2 | EAS 트리거 + web3 훅·서비스 + API 연동 |
> | Phase 3 | web3 UI 통합 + QA |

### Phase 0 — 사전 준비 

**범위**: 팀 전체 블로커 해소 + 인프라 초기 셋업

> 이 Phase는 이후 모든 Phase의 선행 조건을 일괄 해소하기 위한 준비 단계입니다.
> Phase 0 산출물이 완료되어야 Phase 1이 블로커 없이 시작 가능합니다.

- ✅ **[C]** JWT 공유 명세 확정·팀 공유 (secret key, HS256, 만료 24h) — B-1 CommonModule Auth Guard 구현의 선행 조건
- ✅ **[C]** SIWE 인증 흐름 설계 완료 (nonce 발급·서명 검증·JWT 발행 방식 확정)
- ✅ **[C]** tokenURI 스키마 초안 확정 — B-2 메타데이터 JSON 조립 기준 제공
- ✅ **[PM]** EAS 스키마 정의서 작성 (EAS_EXP_TIME / EAS_FAITH_ATT / EAS_WORK_COMP / EAS_EXTRA_ACC 전체 필드)
- ✅ **[PM]** Supabase 프로젝트 초기 설정 (Schema.md 기반 전체 테이블 생성 + 팀 키 공유)
- ✅ **[A]** 와이어프레임 작성 (각 화면의 API 응답 필드 명세 포함 → B-1·B-2 전달용)

**담당**: C(JWT 명세·SIWE 설계·tokenURI 초안) + PM(EAS 스키마·인프라 셋업) + A(와이어프레임)

### Phase 1 — 백엔드 + 인증 + UI 병렬 개발 (4~6주)

**범위**: 백엔드 API 구현 + 블록체인 컨트랙트·인증 + 일반 UI 화면 (mock 데이터 기반)

> web3가 불필요한 화면(근태·인사·스케줄 탭)을 이 단계에서 완성하고, Phase 2에서 실 API를 연동합니다.

- ✅ **[B-1]** QR 출퇴근 시스템 + GPS 검증 (매장 GPS ↔ 스캔 GPS 허용 반경 50m 대조, 불일치 시 거부)
- ✅ **[B-1]** 근태 원장 DB 스키마 + 근태 판정 배치 스케줄러 (결근·지각·정상 자동 판정)
- ✅ **[B-1]** 월간 스케줄 자동 생성 스케줄러 (매월 25일 cron — 반복 근무 패턴 기반 익월 schedules 일괄 생성)
- ✅ **[B-1]** 추가 근무 요청 API (사장님 요청 생성·알바생 신청·사장님 수락) + CommonModule 구현
- ✅ **[B-2]** 사용자·매장·인사 도메인 DB 스키마 + API
- ✅ **[B-2]** 승급 판독 로직 기초 (`/level-check` API 인터페이스)
- ✅ **[C]** SBT 컨트랙트 (ERC-5192: mint/burn only, 양도 불가) + Sepolia 배포
- ✅ **[C]** Multisig 로직 (EIP-712 + `ecrecover` 검증)
- ✅ **[C]** SIWE 인증 구현 완성 (nonce 발급·서명 검증·JWT 발행 + Lv.1 자동 민팅)
- ✅ **[A]** 알바생 일반 UI 화면 구현 (mock 데이터): QR 스캔 탭, 근태 관리 탭, 추가 근무 탭, 매장 연결 섹션

**담당**: B-1(근태·QR) + B-2(사용자·매장·승급 기초) + C(컨트랙트·인증) + A(일반 UI mock)

### Phase 2 — 연동 + EAS + web3 훅·서비스 (3~4주)

**범위**: EAS 자동 발행 트리거 + A의 mock → 실 API 연동 + C의 web3 훅·서비스 레이어 완성

> A는 Phase 1에서 구현한 mock UI를 Phase 2에서 완성된 B-1·B-2 API에 연동합니다.
> C는 Phase 3에서 A가 사용할 web3 훅·서비스 인터페이스를 이 단계에서 확정·공유합니다.

- ✅ **[PM]** EAS 발급 트리거 구현 (Supabase Edge Functions: 조건 달성 시 자동 발행)
- ✅ **[PM]** EAS 발급 실패 재시도 로직 (최대 3회 자동 재시도)
- ✅ **[PM]** 승급 판독 트리거 (EAS 발급 완료 이벤트 → B-2 `/level-check` 호출)
- ✅ **[B-2]** 뱃지 이미지 Supabase Storage 업로드 (PM 원본 이미지 수령 후)
- ✅ **[B-2]** SBT 메타데이터 JSON 조립 → Storage 저장 → tokenURI 연결
- ✅ **[B-2]** 승급 판독 로직 완성 + Multisig 중계 (`/approvals/sign` API)
- ✅ **[C]** web3 훅·서비스 레이어 구현 및 인터페이스 명세 A에게 전달 (Phase 3 시작 전 선행 조건)
  - `useSBTData()` 훅 (알바생 경력 대시보드용)
  - EIP-712 서명 서비스 (사장님 승급 승인용)
  - QR 스캔 + EAS 조회 서비스 (사장님 지원자 검증용)
- ✅ **[A]** Phase 1 mock UI → B-1·B-2 실 API 연동 (출퇴근, 근태, 인사, 매장 관리 등)
- ✅ **[A]** 지갑 연동 UI + 추가 정보 입력 화면 + 로그인 라우팅 (C 모듈 연동)

**담당**: PM(EAS 트리거) + B-2(뱃지·메타데이터·Multisig 완성) + C(web3 훅·서비스) + A(API 연동)

### Phase 3 — web3 UI 통합 + QA (3~4주)

**범위**: C의 훅·서비스 기반 web3 UI 컴포넌트 구현 + 전체 E2E 검증

> C가 Phase 2에서 확정한 인터페이스 명세를 기반으로 A가 web3 UI 컴포넌트를 구현합니다.

- ✅ **[A]** 알바생 경력 프로필 대시보드 (SBT 배지 그리드, 레벨업 프로그레스 바, 포트폴리오 공유 QR) — `useSBTData()` 연동
